# patient-profile Feature Specification

## Summary
The Patient Profile module provides a logged-in patient with the ability to view and update their identity and clinical context data stored in a dedicated patients table. It exposes typed REST endpoints secured by session-based identity checks, a shared ts-rest/Zod contract consumed by downstream modules, and a fully tested React UI with optimistic updates. The module is a self-contained vertical slice that serves as the canonical source of patient profile data for the broader application.

## Functional Requirements
1. **FR-001** GET /api/patient/profile returns HTTP 200 with the authenticated patient's profile fields (firstName, lastName, dateOfBirth, allergies, chronicConditions, regularMedications); identity is derived exclusively from the authenticated session, never from a URL parameter.
2. **FR-002** PUT /api/patient/profile accepts a full update payload, persists all editable fields (firstName, lastName, dateOfBirth, allergies, chronicConditions, regularMedications) to the patients table, and returns HTTP 200 with the updated profile; subsequent GET calls within the same and future sessions reflect the new values.
3. **FR-003** The patients table uses user_id as both primary key and foreign key referencing users.id with ON DELETE CASCADE, ensuring a deleted user record automatically removes the associated patient profile row.
4. **FR-004** A shared ts-rest contract (patientProfileContract) with Zod schemas is published from packages/contracts and consumed identically by apps/api route handlers and the apps/web React client, eliminating any manual type duplication.
5. **FR-005** The /patient/profile React page renders a single form with controlled text inputs for identity fields and multi-line textareas for clinical context fields; the form is pre-populated via TanStack Query from GET /api/patient/profile on mount.
6. **FR-006** On form submission the UI applies an optimistic update immediately, disables the save button for the duration of the in-flight PUT request, and shows a success toast on HTTP 200 or restores the previous values and displays a field-level error message on any non-2xx response.
7. **FR-007** Server-side validation errors from PUT /api/patient/profile are returned as HTTP 422 with a JSON body containing a per-field errors map; the React form surfaces each error message adjacent to the offending input field without a full page reload.

## Non-Functional Requirements
- **NFR-001** GET /api/patient/profile and PUT /api/patient/profile must complete with a p95 response time under 200 ms when the database has up to 10 000 patient rows and the request is served from a co-located Node.js process and PostgreSQL 16 instance.
- **NFR-002** All API routes must be covered by Vitest integration tests and all React form interactions must be covered by Playwright e2e tests; pnpm verify (typecheck + test + lint) must exit 0 before any PR is opened.
- **NFR-003** The module must not log or persist any field value in plaintext application logs at severity INFO or above; structured error logs must omit PII fields from the request body.

## Edge Cases
- **EC-001** Concurrent PUT requests from the same authenticated patient session: the second request arriving while the first is still being processed must not produce a partial write; the database row must reflect exactly one of the two complete payloads, and both responses must be valid HTTP 200 or 422 — no 500 or data corruption is acceptable.
- **EC-002** Patient row does not yet exist in the patients table when GET /api/patient/profile is called (e.g., user was created via a path that skipped profile seeding): the API must return HTTP 404 with body {"error":"profile not found"} rather than a 500 or an empty object.
- **EC-003** dateOfBirth field submitted as exactly today's date (current UTC date): the API must reject it with HTTP 422 and error message "dateOfBirth must be a date in the past" because the constraint requires a date strictly before today.
- **EC-004** PUT /api/patient/profile payload where allergies, chronicConditions, or regularMedications is an empty string: the API must accept the value and persist it as an empty string (not null), and the subsequent GET must return the same empty string, preserving the patient's explicit intent to clear the field.

## Validation Rules
- **VR-001** firstName and lastName are required string fields with a minimum length of 1 and a maximum length of 100 characters. A PUT request with either field absent, null, an empty string, or exceeding 100 characters must be rejected with HTTP 422 and a JSON body containing {"errors":{"<fieldName>":"<fieldName> is required and must be between 1 and 100 characters"}}.
- **VR-002** dateOfBirth must be a valid ISO 8601 date string (YYYY-MM-DD) representing a calendar date strictly before the current UTC date. A PUT request with a malformed date string, a future date, or today's date must be rejected with HTTP 422 and error message "dateOfBirth must be a valid date in the past".
- **VR-003** allergies, chronicConditions, and regularMedications are optional string fields with a maximum length of 2000 characters each. A PUT request with any of these fields exceeding 2000 characters must be rejected with HTTP 422 and error message "<fieldName> must not exceed 2000 characters".
- **VR-004** Any PUT /api/patient/profile request body containing unrecognized keys beyond the defined contract fields must be rejected with HTTP 422 and error message "Request body contains unknown fields" to prevent mass-assignment vulnerabilities.

## Error Handling
- **EH-001** An authenticated user with role doctor sends GET or PUT to /api/patient/profile: HTTP 403 Forbidden with body {"error":"Forbidden"}. No profile data is read or written. No rollback is needed because no write was attempted.
- **EH-002** An authenticated patient sends PUT /api/patient/profile but the database write fails due to a transient PostgreSQL connection error or constraint violation not covered by application-layer validation: HTTP 500 Internal Server Error with body {"error":"Internal server error"}. The database transaction is rolled back automatically; the patients row retains its last successfully committed state. A structured error is emitted to the server log at ERROR severity (without PII field values).
- **EH-003** An unauthenticated request (missing or expired session cookie) reaches GET or PUT /api/patient/profile: HTTP 401 Unauthorized with body {"error":"Unauthorized"}. No profile data is accessed. The session is not modified.
- **EH-004** An authenticated patient sends GET /api/patient/profile but no row exists in the patients table for their user_id: HTTP 404 Not Found with body {"error":"profile not found"}. No write or rollback occurs. The client UI must display a prompt to complete profile setup rather than rendering an empty form silently.

## Authorization
- **AUTH-001** Role patient (as encoded in the authenticated session) is the only role permitted to call GET /api/patient/profile or PUT /api/patient/profile. Identity for both read and write is taken exclusively from the session; a patient cannot supply a different user_id in the request body or URL to access another patient's record — any such attempt returns HTTP 403.
- **AUTH-002** Role doctor is explicitly denied access to GET /api/patient/profile and PUT /api/patient/profile. Any request from a session with role doctor must be rejected with HTTP 403 Forbidden regardless of whether a valid patient profile exists for that user_id.
- **AUTH-003** Unauthenticated callers (no valid session) are denied access to GET /api/patient/profile and PUT /api/patient/profile with HTTP 401 Unauthorized; the authorization check for role must occur only after session validity is confirmed, never before.

## Acceptance Criteria
- [ ] **AC-001** Given an authenticated patient with a valid session and an existing row in the patients table, when they send GET /api/patient/profile, then the server returns HTTP 200 with a JSON body containing the fields firstName, lastName, dateOfBirth, allergies, chronicConditions, and regularMedications matching the values stored for that patient's user_id.
- [ ] **AC-002** Given an authenticated patient with a valid session, when they send GET /api/patient/profile with a different user_id supplied as a query parameter or in the request body, then the server ignores that parameter and returns HTTP 200 with only the profile data associated with the session's own user_id.
- [ ] **AC-003** Given an authenticated patient with a valid session and an existing patients row, when they send PUT /api/patient/profile with a valid payload containing updated values for firstName, lastName, dateOfBirth, allergies, chronicConditions, and regularMedications, then the server returns HTTP 200 with a JSON body reflecting all updated field values.
- [ ] **AC-004** Given an authenticated patient who has successfully completed a PUT /api/patient/profile request with updated field values, when they end their session and then authenticate again and send GET /api/patient/profile, then the server returns HTTP 200 with the field values that were submitted in the previous PUT request, confirming persistence across sessions.
- [ ] **AC-005** Given a patients table row exists for a given user_id, when the corresponding row in the users table is deleted, then the patients table row for that user_id is automatically removed via ON DELETE CASCADE and no orphaned patient profile record remains.
- [ ] **AC-006** Given the patients table schema, when inspecting the table definition, then user_id is both the primary key of the patients table and a foreign key referencing users.id with ON DELETE CASCADE configured at the database level.
- [ ] **AC-007** Given the packages/contracts package exports a patientProfileContract defined with ts-rest and Zod schemas, when apps/api route handlers and apps/web React client both import patientProfileContract from packages/contracts, then both sides share identical TypeScript types and Zod validation logic without any manually duplicated type definitions.
- [ ] **AC-008** Given the patientProfileContract Zod schema defines validation rules for all editable fields, when apps/api applies that schema to an incoming PUT /api/patient/profile request body, then the same schema constraints used in the contract are enforced server-side without re-implementing separate validation logic.
- [ ] **AC-009** Given an authenticated patient navigates to /patient/profile in the browser, when the page finishes loading, then a single form is rendered containing controlled text inputs for firstName and lastName, a controlled date input for dateOfBirth, and multi-line textareas for allergies, chronicConditions, and regularMedications, each pre-populated with the values returned by GET /api/patient/profile.
- [ ] **AC-010** Given an authenticated patient is on the /patient/profile page, when the TanStack Query fetch of GET /api/patient/profile completes successfully, then every form field displays the exact value stored in the patient's profile without requiring any manual user interaction.
- [ ] **AC-011** Given an authenticated patient has made changes to their profile form, when they click the save button, then the UI immediately applies an optimistic update reflecting the new values, the save button becomes disabled for the duration of the in-flight PUT request, and upon receiving HTTP 200 a success toast notification is displayed.
- [ ] **AC-012** Given an authenticated patient submits the profile form and the PUT /api/patient/profile request returns a non-2xx response, when the response is received, then the form fields revert to the values they held before the submission, the save button is re-enabled, and a field-level error message is displayed adjacent to the offending input field.
- [ ] **AC-013** Given an authenticated patient sends PUT /api/patient/profile with a payload that fails server-side Zod validation, when the API processes the request, then it returns HTTP 422 with a JSON body of the form {"errors":{"<fieldName>":"<errorMessage>"}} containing an entry for each invalid field.
- [ ] **AC-014** Given an authenticated patient submits the profile form with a value that triggers a server-side validation error, when the HTTP 422 response is received by the React client, then an error message is rendered directly adjacent to the input field identified by the errors map key, and the page does not perform a full reload.
