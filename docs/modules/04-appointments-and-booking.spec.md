# appointments-and-booking Feature Specification

## Summary
This module implements the end-to-end appointment booking lifecycle, allowing patients to browse doctors, select available slots, and atomically book them while preventing double-booking via database-level locking. It provides role-scoped list and detail views for both patients and doctors, with the doctor's detail panel composing data from the patient profile and document-sharing modules. The feature exists to deliver the core scheduling workflow and the doctor's before-visit information panel as defined by the product spec.

## Functional Requirements
1. **FR-001** Patient can list doctors filtered by specialization keyword: GET /doctors?specialization=\<keyword\> returns only doctors whose specialization contains the keyword (case-insensitive substring match), returning HTTP 200 with an array of doctor objects including id, name, and specialization.
2. **FR-002** Patient can view upcoming free slots for a selected doctor: GET /doctors/:doctorId/slots returns HTTP 200 with an array of slot objects (id, datetime, status) where status equals 'free' and datetime is strictly in the future relative to the server's current UTC time.
3. **FR-003** Patient books a free slot atomically: POST /appointments with body {slotId} acquires a SELECT FOR UPDATE lock on the slot row, inserts a new appointment record with status 'scheduled', and updates the slot status to 'booked' within a single database transaction, returning HTTP 201 with the created appointment object.
4. **FR-004** Concurrent double-booking of the same slot results in exactly one HTTP 201 success and one HTTP 409 conflict response; the second transaction that encounters the already-booked slot must rollback and return HTTP 409 with error code SLOT_ALREADY_BOOKED without inserting a duplicate appointment record.
5. **FR-005** Patient can view their own appointments: GET /appointments returns HTTP 200 with an array of appointment objects including doctorName, doctorSpecialization, slotDatetime, and status, split into upcoming (slotDatetime >= now) and past (slotDatetime < now) groups.
6. **FR-006** Doctor can view their schedule: GET /doctor/appointments returns HTTP 200 with appointment objects grouped by date (ISO 8601 date string keys), each containing patientName and slotDatetime, covering both upcoming and past appointments.
7. **FR-007** Doctor can view appointment detail panel: GET /doctor/appointments/:appointmentId returns HTTP 200 with a composite object containing appointment metadata, the patient's profile sourced from Module 01 (name, contact, medical history summary), and the list of documents the patient shared with the doctor from Module 02 including per-document download links.
8. **FR-008** The appointments.status field is only ever written as 'scheduled' by this module; any code path in this module that would write a different status value must not exist — status transitions to 'completed' are owned exclusively by Module 05.

## Non-Functional Requirements
- **NFR-001** The POST /appointments booking endpoint must complete end-to-end (including database transaction commit) in under 500 ms at p95 under a load of 50 concurrent booking requests targeting the same slot, as measured by a load test run in the CI integration environment.
- **NFR-002** All GET list endpoints (doctor list, slot list, patient appointment list, doctor schedule) must respond in under 200 ms at p95 for data sets up to 10,000 rows in the relevant tables, with appropriate database indexes on foreign keys and status/datetime columns.
- **NFR-003** The booking transaction must be idempotent from the caller's perspective: if the network drops after commit but before the response is sent, a retry with the same slotId from the same patient must return HTTP 409 SLOT_ALREADY_BOOKED rather than creating a duplicate appointment.
- **NFR-004** Every HTTP 4xx and 5xx response from this module must include a structured JSON body with at minimum the fields 'code' (string error code) and 'message' (human-readable description) to enable consistent frontend error handling and log correlation.

## Edge Cases
- **EC-001** Sequential double-booking: if patient A books slot S successfully (HTTP 201) and then patient B attempts to book the same slot S in a separate serial request, the second request must return HTTP 409 with code SLOT_ALREADY_BOOKED; no second appointment record may exist for slot S in the database.
- **EC-002** Booking a slot whose datetime is in the past: if a patient submits POST /appointments with a slotId whose datetime is before the server's current UTC time, the endpoint must return HTTP 422 with code SLOT_IN_PAST and must not insert an appointment record or modify the slot row.
- **EC-003** Doctor list filtered by specialization keyword with zero matches: GET /doctors?specialization=nonexistent returns HTTP 200 with an empty array [], not HTTP 404.
- **EC-004** Patient views appointments when they have none: GET /appointments for a patient with no appointment records returns HTTP 200 with body {upcoming: [], past: []} rather than HTTP 404 or an error.
- **EC-005** Doctor detail panel when Module 01 or Module 02 returns a non-200 response for cross-module data: GET /doctor/appointments/:appointmentId must still return HTTP 200 with appointment metadata present, substituting null for the unavailable sub-resource (patientProfile: null or sharedDocuments: null) and must not propagate the upstream error as a 5xx.

## Validation Rules
- **VR-001** POST /appointments request body must contain slotId as a non-empty UUID v4 string; if slotId is absent, not a string, or not a valid UUID v4, the endpoint returns HTTP 422 with code VALIDATION_ERROR and message 'slotId must be a valid UUID'.
- **VR-002** GET /doctors?specialization query parameter, when provided, must be a non-empty string of 1–100 characters; if it is an empty string or exceeds 100 characters the endpoint returns HTTP 422 with code VALIDATION_ERROR and message 'specialization must be between 1 and 100 characters'.
- **VR-003** GET /doctor/appointments/:appointmentId — the appointmentId path parameter must be a valid UUID v4; if it is not, the endpoint returns HTTP 422 with code VALIDATION_ERROR and message 'appointmentId must be a valid UUID' without performing any database query.
- **VR-004** POST /appointments must reject requests where the resolved slotId belongs to a slot whose status is not 'free' at the time the SELECT FOR UPDATE lock is acquired; the response must be HTTP 409 with code SLOT_ALREADY_BOOKED and message 'The requested slot is no longer available'.

## Error Handling
- **EH-001** The database transaction for booking fails due to a deadlock or serialization error (PostgreSQL error code 40001 or 40P01): HTTP 503 with code BOOKING_TRANSACTION_FAILED and message 'Booking could not be completed due to a transient conflict; please retry'. The transaction is fully rolled back; no appointment record is inserted and the slot status remains unchanged.
- **EH-002** POST /appointments is called with a slotId that does not exist in the slots table: HTTP 404 with code SLOT_NOT_FOUND and message 'Slot not found'. No database write occurs.
- **EH-003** Doctor attempts GET /doctor/appointments/:appointmentId for an appointmentId that exists in the database but belongs to a different doctor: HTTP 403 with code FORBIDDEN and message 'You do not have access to this appointment'. No patient profile or document data is fetched from Module 01 or Module 02.
- **EH-004** An unhandled exception or database connection error occurs inside the booking transaction: HTTP 500 with code INTERNAL_ERROR and message 'An unexpected error occurred'. The database transaction is rolled back via the connection pool's error handler; no partial appointment record or slot status mutation persists.

## Authorization
- **AUTH-001** Role PATIENT is allowed to: list doctors (GET /doctors), list slots for a doctor (GET /doctors/:doctorId/slots), book a slot (POST /appointments), and view only their own appointments (GET /appointments). A PATIENT is denied access to any /doctor/* endpoint and receives HTTP 403 with code FORBIDDEN.
- **AUTH-002** Role DOCTOR is allowed to: view their own schedule (GET /doctor/appointments) and view detail for their own appointments (GET /doctor/appointments/:appointmentId). A DOCTOR is denied access to POST /appointments and to GET /appointments (patient list endpoint) and receives HTTP 403 with code FORBIDDEN.
- **AUTH-003** Unauthenticated requests (missing or invalid JWT) to any endpoint in this module receive HTTP 401 with code UNAUTHORIZED and message 'Authentication required'; no data is returned.
- **AUTH-004** A PATIENT may only read their own appointment records: GET /appointments filters results by the authenticated patient's userId at the query level; a patient cannot retrieve appointments belonging to another patient even by manipulating query parameters.

## Acceptance Criteria
- [ ] **AC-001** Given a patient with a valid JWT, when they send GET /doctors?specialization=cardio, then the server returns HTTP 200 with a JSON array containing only doctor objects whose specialization field contains 'cardio' (case-insensitive), each object including id, name, and specialization fields.
- [ ] **AC-002** Given a patient with a valid JWT, when they send GET /doctors?specialization=CARDIO (uppercase), then the server returns HTTP 200 with a JSON array that includes doctors whose specialization contains 'cardio' in any casing, confirming case-insensitive substring matching.
- [ ] **AC-003** Given a patient with a valid JWT, when they send GET /doctors?specialization=nonexistent, then the server returns HTTP 200 with an empty array [].
- [ ] **AC-004** Given a patient with a valid JWT and a doctorId referencing an existing doctor, when they send GET /doctors/:doctorId/slots, then the server returns HTTP 200 with a JSON array of slot objects each containing id, datetime, and status fields, where every returned slot has status equal to 'free' and a datetime strictly greater than the server's current UTC time.
- [ ] **AC-005** Given a patient with a valid JWT and a doctorId whose only slots are either already booked or have datetimes in the past, when they send GET /doctors/:doctorId/slots, then the server returns HTTP 200 with an empty array [].
- [ ] **AC-006** Given a patient with a valid JWT and a slotId referencing a free slot with a future datetime, when they send POST /appointments with body {"slotId": "\<slotId\>"}, then the server returns HTTP 201 with a JSON object representing the created appointment, the slot row's status field is updated to 'booked' in the database, and a new appointment record with status 'scheduled' exists in the database, all within a single committed database transaction.
- [ ] **AC-007** Given a patient with a valid JWT and a slotId referencing a free slot with a future datetime, when they send POST /appointments with body {"slotId": "\<slotId\>"}, then the booking endpoint acquires a SELECT FOR UPDATE lock on the slot row before inserting the appointment record, ensuring the lock, insert, and slot status update all occur within the same database transaction.
- [ ] **AC-008** Given two patients each holding a valid JWT, when both concurrently send POST /appointments with the same slotId at the same time, then exactly one request receives HTTP 201 with the created appointment object and exactly one request receives HTTP 409 with code 'SLOT_ALREADY_BOOKED', and only one appointment record exists in the database for that slot.
- [ ] **AC-009** Given a patient with a valid JWT who successfully books slot S (receiving HTTP 201), when a second patient with a valid JWT sends POST /appointments with the same slotId S, then the server returns HTTP 409 with code 'SLOT_ALREADY_BOOKED' and no second appointment record exists in the database for slot S.
- [ ] **AC-010** Given a patient with a valid JWT who has both upcoming and past appointments, when they send GET /appointments, then the server returns HTTP 200 with a JSON object containing an 'upcoming' array of appointments whose slotDatetime is greater than or equal to the current UTC time and a 'past' array of appointments whose slotDatetime is less than the current UTC time, each appointment object including doctorName, doctorSpecialization, slotDatetime, and status fields.
- [ ] **AC-011** Given a patient with a valid JWT who has no appointment records, when they send GET /appointments, then the server returns HTTP 200 with body {"upcoming": [], "past": []}.
- [ ] **AC-012** Given two patients each with a valid JWT and each having their own appointments, when patient A sends GET /appointments, then the server returns HTTP 200 with only appointment records belonging to patient A and no appointment records belonging to patient B.
- [ ] **AC-013** Given a doctor with a valid JWT who has appointments on multiple distinct dates, when they send GET /doctor/appointments, then the server returns HTTP 200 with a JSON object whose keys are ISO 8601 date strings and whose values are arrays of appointment objects each containing patientName and slotDatetime, covering both upcoming and past appointments.
- [ ] **AC-014** Given a doctor with a valid JWT who has no appointments, when they send GET /doctor/appointments, then the server returns HTTP 200 with an empty JSON object {}.
- [ ] **AC-015** Given a doctor with a valid JWT and an appointmentId that belongs to that doctor, when they send GET /doctor/appointments/:appointmentId and both Module 01 and Module 02 return successful responses, then the server returns HTTP 200 with a composite JSON object containing appointment metadata, a patientProfile object sourced from Module 01 (including name, contact, and medical history summary), and a sharedDocuments array sourced from Module 02 with per-document download links.
- [ ] **AC-016** Given a doctor with a valid JWT and an appointmentId that belongs to that doctor, when they send GET /doctor/appointments/:appointmentId and Module 01 returns a non-200 response, then the server returns HTTP 200 with appointment metadata present and patientProfile set to null, without propagating the upstream error as a 5xx.
- [ ] **AC-017** Given a doctor with a valid JWT and an appointmentId that belongs to that doctor, when they send GET /doctor/appointments/:appointmentId and Module 02 returns a non-200 response, then the server returns HTTP 200 with appointment metadata present and sharedDocuments set to null, without propagating the upstream error as a 5xx.
- [ ] **AC-018** Given a patient with a valid JWT and a free future slot, when they send POST /appointments with body {"slotId": "\<slotId\>"} and the booking succeeds, then the newly created appointment record in the database has a status field value of exactly 'scheduled' and no code path in this module writes any other status value to the appointments.status field.
