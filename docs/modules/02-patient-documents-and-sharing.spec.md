# patient-documents-and-sharing Feature Specification

## Summary
This feature implements end-to-end medical document management for a patient–doctor platform, covering patient-controlled upload, listing, deletion, and fine-grained per-doctor sharing entirely decoupled from appointments. It exposes role-guarded API endpoints, a Drizzle-backed schema with cascade deletes, and a reusable React component consumed by the Appointments module (Module 04). The goal is to give patients direct control over document visibility while ensuring doctors can only access files they have been explicitly granted permission to view.

## Functional Requirements
1. **FR-001** — A patient can upload a medical document via a multipart/form-data POST endpoint; the server stores the file under apps/api/storage/documents/ using a UUID filename, persists a row in the documents table with original filename, mime type, byte size, and ISO-8601 upload timestamp, and returns HTTP 201 with the new document's metadata in the response body.
2. **FR-002** — A patient can retrieve a list of all their own documents via a GET endpoint that returns HTTP 200 with a JSON array where each element contains at minimum: id, filename, mimeType, size (bytes), and uploadedAt (ISO-8601); the list is ordered by uploadedAt descending.
3. **FR-003** — A patient can delete a document via a DELETE endpoint; the system atomically removes the database row, all associated document_shares rows, and the file from disk within a single transaction, and returns HTTP 204 on success.
4. **FR-004** — A patient can retrieve the sharing state of a document via a GET endpoint that returns HTTP 200 with a JSON array listing every doctor in the system, each entry containing doctorId, displayName, and a boolean hasAccess field indicating whether the document is currently shared with that doctor.
5. **FR-005** — A patient can grant access to a specific doctor for a specific document via a PUT/POST endpoint; on success the system inserts a document_shares row and returns HTTP 200; subsequent doctor download requests for that document return HTTP 200.
6. **FR-006** — A patient can revoke access from a specific doctor for a specific document via a DELETE endpoint targeting the sharing record; on success the system removes the document_shares row and returns HTTP 200; subsequent doctor download requests for that document return HTTP 403.
7. **FR-007** — A doctor can retrieve a 'Shared with me' list via a GET endpoint that returns HTTP 200 with documents grouped by patient, where each group includes patientId, patientDisplayName, and an array of document metadata objects (id, filename, mimeType, size, uploadedAt).
8. **FR-008** — A doctor can download a document that has been explicitly shared with them via a GET endpoint; the server streams the file binary content using chunked transfer encoding without buffering the entire file in memory and sets Content-Type to the stored mimeType and Content-Disposition to attachment with the original filename.
9. **FR-009** — The frontend exposes a reusable React component and its associated TanStack Query hook under a named export from this module; Module 04 (appointments) imports and renders this component without reimplementing any doctor-side shared-document logic.

## Non-Functional Requirements
- **NFR-001** — Document list and sharing-state GET endpoints must respond in under 200 ms at p95 under a load of 50 concurrent requests against a database with 10 000 document rows.
- **NFR-002** — File download must begin streaming the first byte within 500 ms of the request being accepted for files up to 10 MB, measured at p95 on the local docker-compose dev environment.
- **NFR-003** — Stored filenames on disk must be Version 4 UUIDs with no extension; the original filename must never appear in the filesystem path, preventing path enumeration attacks.
- **NFR-004** — No document file may be served via a public static path; every download request must pass the authorization check before any byte of file content is written to the response.
- **NFR-005** — pnpm verify (typecheck + lint + Vitest unit/integration tests) must exit with code 0 before any PR is raised; test coverage for the document service layer must be 100% of branches.

## Edge Cases
- **EC-001** — When a patient uploads a file whose byte size is exactly 10 485 760 bytes (10 MiB), the server must accept it and return HTTP 201; when the file is 10 485 761 bytes the server must reject it with HTTP 413.
- **EC-002** — When a patient attempts to delete a document that is currently shared with one or more doctors, the system must atomically remove both the document row and all document_shares rows in the same transaction, then delete the file from disk; the delete must return HTTP 204 and no orphan sharing records may remain.
- **EC-003** — When a patient calls the grant-access endpoint for a doctor that already has access to that document (duplicate share), the server must return HTTP 200 without creating a duplicate document_shares row (upsert semantics).
- **EC-004** — When a patient calls the revoke-access endpoint for a doctor that does not currently have access to the document, the server must return HTTP 200 (idempotent) without error.
- **EC-005** — When a patient has zero documents, the list endpoint must return HTTP 200 with an empty JSON array, not HTTP 404 or null.
- **EC-006** — When a doctor has no documents shared with them, the 'Shared with me' endpoint must return HTTP 200 with an empty JSON array, not HTTP 404 or null.

## Validation Rules
- **VR-001** — Uploads with a Content-Type (or file part mime type) not in [application/pdf, image/png, image/jpeg, image/webp] must be rejected with HTTP 415 and a JSON body { "error": "UNSUPPORTED_MEDIA_TYPE", "message": "Allowed types: application/pdf, image/png, image/jpeg, image/webp" }.
- **VR-002** — Uploads where the file part exceeds 10 485 760 bytes (10 MiB) must be rejected with HTTP 413 and a JSON body { "error": "FILE_TOO_LARGE", "message": "Maximum upload size is 10 MB" }; the partial file must not be persisted to disk or inserted into the database.
- **VR-003** — A request to grant or revoke document sharing where the doctorId path parameter does not correspond to a user with the doctor role must be rejected with HTTP 404 and a JSON body { "error": "DOCTOR_NOT_FOUND", "message": "No doctor found with the given id" }.
- **VR-004** — A request referencing a documentId that does not exist in the documents table must be rejected with HTTP 404 and a JSON body { "error": "DOCUMENT_NOT_FOUND", "message": "No document found with the given id" }.

## Error Handling
- **EH-001** — A doctor requests download of a document that has not been explicitly shared with them (no matching document_shares row for that doctorId and documentId): HTTP 403 with JSON body { "error": "ACCESS_DENIED", "message": "You do not have permission to access this document" }. No file bytes are written to the response. No rollback required as no state was mutated.
- **EH-002** — The filesystem delete call fails after the database row and sharing records have already been removed within the transaction (partial delete failure): HTTP 500 with JSON body { "error": "DELETE_FAILED", "message": "Document metadata removed but file could not be deleted from storage" }. The database transaction is committed (orphan-file recovery is out of scope per constraints). The response body must indicate the partial state so operators can manually clean up.
- **EH-003** — The multipart file write to apps/api/storage/documents/ fails mid-upload due to a disk I/O error: HTTP 500 with JSON body { "error": "STORAGE_ERROR", "message": "Failed to persist file to storage" }. Any partially written file is deleted before responding. No database row is inserted.
- **EH-004** — A request body is missing the required file part in a multipart upload: HTTP 422 with JSON body { "error": "MISSING_FILE", "message": "Request must include a file part" }. No file is written and no database row is inserted.

## Authorization
- **AUTH-001** — A user with the patient role may call upload (POST), list (GET), delete (DELETE), get-sharing-state (GET), grant-access, and revoke-access endpoints only for documents where documents.patientId equals their own userId; any attempt by a patient to access another patient's documents returns HTTP 403.
- **AUTH-002** — A user with the doctor role may call the 'Shared with me' list endpoint and the document download endpoint only; attempting to call any patient-only endpoint (upload, list own, delete, manage sharing) returns HTTP 403 with JSON body { "error": "FORBIDDEN", "message": "This endpoint requires the patient role" }.
- **AUTH-003** — An unauthenticated request (missing or invalid JWT) to any document endpoint returns HTTP 401 with JSON body { "error": "UNAUTHORIZED", "message": "Authentication required" } before any business logic is executed.
- **AUTH-004** — A user with the doctor role may download a specific document only when a document_shares row exists with documentId matching the requested document and doctorId matching the authenticated user's userId; the authorization check is performed on every request and is never cached client-side.

## Acceptance Criteria
- [ ] **AC-001** — Given a patient with a valid JWT, when they send a multipart/form-data POST to the document upload endpoint with an application/pdf file of 1 MB, then the server returns HTTP 201 with a JSON body containing the document's id, filename, mimeType, size in bytes, and uploadedAt in ISO-8601 format, and a file named as a Version 4 UUID (with no extension and no original filename) is created under apps/api/storage/documents/.
- [ ] **AC-002** — Given a patient with a valid JWT, when the multipart/form-data POST upload completes successfully, then a row is inserted into the documents table containing the original filename exactly as submitted, the detected mime type, the byte size of the file, and an uploadedAt ISO-8601 timestamp, and the file path stored in the row references only the UUID-based disk filename with no original filename component.
- [ ] **AC-003** — Given a patient with a valid JWT who has previously uploaded three documents at distinct timestamps T1 < T2 < T3, when they send a GET request to the list-documents endpoint, then the server returns HTTP 200 with a JSON array of exactly three objects ordered by uploadedAt descending (T3 first, T1 last), each object containing the fields id, filename, mimeType, size, and uploadedAt.
- [ ] **AC-004** — Given a patient with a valid JWT who has uploaded zero documents, when they send a GET request to the list-documents endpoint, then the server returns HTTP 200 with an empty JSON array [] and not HTTP 404 or a null body.
- [ ] **AC-005** — Given a patient with a valid JWT who owns a document that is currently shared with two doctors, when the patient sends a DELETE request to the document endpoint for that document's id, then the server returns HTTP 204, the row for that document no longer exists in the documents table, all document_shares rows referencing that documentId are removed, and the UUID-named file no longer exists under apps/api/storage/documents/.
- [ ] **AC-006** — Given a patient with a valid JWT who owns a document shared with one doctor, when the DELETE request for that document completes with HTTP 204, then there is no observable intermediate state where the documents row is absent but the document_shares row still exists, nor any state where the document_shares row is absent but the documents row still exists, confirming the removal occurred within a single atomic transaction.
- [ ] **AC-007** — Given a patient with a valid JWT who owns a document that is shared with exactly one of three doctors in the system, when they send a GET request to the sharing-state endpoint for that document's id, then the server returns HTTP 200 with a JSON array of three objects, each containing doctorId, displayName, and hasAccess, where hasAccess is true only for the one doctor the document is shared with and false for the other two.
- [ ] **AC-008** — Given a patient with a valid JWT who owns a document and a target doctor who does not currently have access, when the patient sends a PUT or POST to the grant-access endpoint with that doctor's id, then the server returns HTTP 200, a document_shares row exists in the database for that documentId and doctorId, and a subsequent GET download request by that doctor returns HTTP 200.
- [ ] **AC-009** — Given a patient with a valid JWT who owns a document and a target doctor who already has access to that document (a document_shares row already exists), when the patient sends a grant-access request for the same doctorId, then the server returns HTTP 200 and exactly one document_shares row exists for that documentId and doctorId pair — no duplicate row is inserted.
- [ ] **AC-010** — Given a patient with a valid JWT who owns a document and a target doctor who currently has access to that document, when the patient sends a DELETE to the revoke-access endpoint specifying that doctor's id, then the server returns HTTP 200, the document_shares row for that documentId and doctorId is removed from the database, and a subsequent GET download request by that doctor returns HTTP 403 with JSON body { "error": "ACCESS_DENIED", "message": "You do not have permission to access this document" }.
- [ ] **AC-011** — Given a patient with a valid JWT who owns a document and a target doctor who does not currently have access to that document (no document_shares row exists), when the patient sends a DELETE to the revoke-access endpoint specifying that doctor's id, then the server returns HTTP 200 without error and no document_shares row is created or modified.
- [ ] **AC-012** — Given a doctor with a valid JWT who has documents shared by two distinct patients, when they send a GET request to the 'Shared with me' endpoint, then the server returns HTTP 200 with a JSON array of two objects, each containing patientId, patientDisplayName, and a nested array of document metadata objects where each metadata object includes id, filename, mimeType, size, and uploadedAt.
- [ ] **AC-013** — Given a doctor with a valid JWT who has no documents shared with them, when they send a GET request to the 'Shared with me' endpoint, then the server returns HTTP 200 with an empty JSON array [] and not HTTP 404 or a null body.
- [ ] **AC-014** — Given a doctor with a valid JWT and a document that has been explicitly shared with that doctor by the owning patient, when the doctor sends a GET request to the document download endpoint for that document's id, then the server returns HTTP 200 with the file binary content delivered via chunked transfer encoding, the Content-Type header set to the stored mimeType, and the Content-Disposition header set to 'attachment; filename="<original filename>"'.
- [ ] **AC-015** — Given a doctor with a valid JWT and a document that has NOT been shared with that doctor (no document_shares row exists for that doctorId and documentId), when the doctor sends a GET request to the document download endpoint for that document's id, then the server returns HTTP 403 with JSON body { "error": "ACCESS_DENIED", "message": "You do not have permission to access this document" } and zero bytes of file content are written to the response.
- [ ] **AC-016** — Given module-04 (appointments) imports the named React component and TanStack Query hook exported by the documents module without reimplementing any shared-document fetching or rendering logic, when an authenticated doctor navigates to the appointments view that embeds the imported component, then the 'Shared with me' document list renders correctly using only the component and hook imported from the documents module, with no document-management code duplicated inside module-04.
